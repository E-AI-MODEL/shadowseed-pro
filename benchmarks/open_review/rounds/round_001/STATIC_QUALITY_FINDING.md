# Round 001 — Static quality finding

Date: 2026-05-20
Related: #41, #62, #74
Source artifacts checked:

- `benchmarks/open_review/input/hf_batch.json` (12 AG News test items)
- `results/open_review/open_set_seed_output.json` (60 generated seeds)
- `results/open_review/open_set_review_packets.json` (120 pending packets)

## Why a static check

The pending packets in round 001 were generated by `open_set_candidate_adapter` v0.1. Before asking two reviewers to fill in 120 rows, the seed output was checked statically against:

- `docs/02_atomic_seeds.md` — the seed criteria
- `docs/00_shadow_seed_learning_4_6.md` — the canonical 4.6 source, in particular the one-sentence claim and the "minimale werkende definitie van SSL"

The check is intentionally non-human: if the static evidence already shows the seeds cannot meet the criteria, paying for human review on the same artifacts mostly produces predictable rejections.

## What the adapter actually does

`src/shadowseed/benchmark/open_set_candidate_adapter.py` is a regex-and-template generator, not a language model. Per item it fills five slots from a fixed template set:

1. first capitalized token → `"Rol van {entity} in de gebeurtenis."`
2. presence of claim word ("says/claim/announced") → `"Bron van de centrale bewering."` else `"Onderbouwing van de centrale bewering."`
3. presence of time word → `"Tijdlijn van de beschreven gebeurtenis."` else `"Tijdstip van de beschreven gebeurtenis."`
4. presence of group word ("workers/customers/...") → `"Direct geraakte groep in het item."` else `"Betrokken partij buiten de hoofdactor."`
5. presence of number → `"Herkomst van de genoemde cijfers."` elif decision word → `"Reden voor de beschreven beslissing."` else `"Onzekerheid rond de centrale bewering."`

Across the 12 items in this batch the adapter produced only **21 unique seed texts across 60 slots**, with single templates filling up to 10 of 12 items.

## Check against `02_atomic_seeds.md`

| Criterion (§2) | Round 001 output | Verdict |
|---|---|---|
| Eén gap | Each template names one meta-question | formally yes |
| Specifiek (déze tekst, niet willekeurige) | "Onderbouwing van de centrale bewering" fits any news item | no |
| Toetsbaar | Trivially testable ("does the text cite a source? no") | technically yes, content-empty |
| Relevant | "Tijdstip van de gebeurtenis" does not improve answers about a pension fund article | no |
| Niet-triviaal | "Onzekerheid rond de centrale bewering" is a truism | no |

`02_atomic_seeds.md` §3 explicitly lists as forbidden seed shapes:

> meta-categorieën die naar het item verwijzen in plaats van naar de inhoud, zoals "Onderbouwing van de centrale bewering", "Tijdlijn van de gebeurtenis", "Betrokken partij buiten de hoofdactor", "Onzekerheid rond de centrale bewering"

These are the same strings the adapter emits.

## Check against `00_shadow_seed_learning_4_6.md`

| 4.6 element | Round 001 adapter | Verdict |
|---|---|---|
| One-sentence claim: "een taalmodel detecteert..." | Adapter is regex + templates, no model | violates the one-sentence claim |
| Mechanical kernel §1 ("klein, specifiek, controleerbaar, inhoudelijk relevant, niet-triviaal") | Templates are meta-categories | fails on specifiek, relevant, niet-triviaal |
| Minimal working definition step 1 ("kleine gaps detecteren") | Detects meta-questions, not text-specific gaps | fails the essence |
| Layer C ("kan SSL in onbekende teksten kleine, relevante en toetsbare seeds produceren?") | No SSL detection in this lane — only templating | not Layer C evidence in current form |
| Decision rule §2 ("Helpt dit bij open-set/adversarial/behavioral/transfer-validatie?") | Filling in predictable rejections does not advance any of the four | answer is no |

## Conclusion

Round 001 in its current form is:

- valid as an **infrastructure baseline**: the full pipeline (fetch → adapter → packet generation → summary → report) runs end-to-end on real public HF data
- not valid as **Layer C open-set seed-quality evidence**: the candidates do not meet the seed criteria the project sets for itself

This is a finding about the detection layer, not about reviewer skill or about SSL as a whole. The mechanical kernel (manager, Gate, trace/weight) is unaffected.

## Recommended follow-up

Tracked in #74. Two complementary paths:

1. **`open_set_candidate_adapter_v0.2` — text-grounded extraction.** Replace the meta-category templates with item-specific noun-phrase / entity extraction so candidates reference the actual text. Still no taalmodel, but already much closer to §2 criteria. Useful as a stronger baseline.
2. **Taalmodel-detector path (preferred for Layer C).** Implement a separate detector that calls the existing `hf-transformers` backend (already used by the model-benefit suite) with the detection prompt in `docs/05_prompts.md`. This is the route that actually satisfies the 4.6 one-sentence claim.

Round 002 should be (re-)dispatched only after one of these lands and after the input batch is spread across the four AG News classes (current batch is 11 of 12 in Sci/Tech).

## What stays in the repo

- the pending packets, CSVs and pipeline artifacts under `benchmarks/open_review/` and `results/open_review/` — kept as infrastructure baseline
- this finding document — the formal record that the static check happened and what it showed
- no human review verdicts in the JSON or CSVs — left at `pending` deliberately, so a future Layer-C round on better candidates does not have to disentangle two evidence levels
